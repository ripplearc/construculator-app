Overview

This PR adds a single CLAUDE.md file (68 lines) at the repository root, providing behavioral guidance for AI agents. It covers project commands, CI trigger syntax, Clean Architecture layout, DI/routing conventions, bloc-per-action pattern, Either error handling, enforced lint rules, staged (currently disabled) conventions, and logging. This is a documentation-only change with no runtime impact.

What's Working Well

  • Clear intent separation: The file explicitly opens with "behavioral guidance for agents, not architecture documentation" — sets correct expectations and prevents scope creep into the file over time.
  • Enforced vs. unenforced lint split High value: Separating currently-disabled conventions (CA-627, CA-641, CA-642, CA-633) from hard-enforced rules is the most important part of this file. It prevents agents from incorrectly blocking changes on rules the linter doesn't yet enforce, and gives a clear path for when they will be enforced.
  • Bloc-per-action pattern: Concisely explained with concrete examples (add / delete / rename / change_lock / list). Agents will generate correctly granular blocs rather than monolithic screen-blocs.
  • YouTrack ticket references: Each unenforced convention links a ticket (e.g. CA-642) — keeps the guidance traceable and makes it easy to find the cleanup context.
  • Lean and focused: Follows its own stated principle — doesn't balloon into a full architecture doc. 68 lines is well-calibrated.

Issues & Suggestions

Moderate Contradictory DI guidance

The Hard Rules section says:

No direct instantiation of injectable types — resolve via DI / Modular.get.

But the Conventions section says:

No Modular.get outside a module — constructor-inject into blocs/usecases instead (CA-627).

An agent following the hard rule literally could call Modular.get inside a bloc — exactly what the convention prohibits. Consider rewording the hard rule to:

No direct instantiation of injectable types — wire dependencies through the module's binds() and constructor-inject. Never call Modular.get inside blocs or usecases.

This collapses both statements into one unambiguous instruction.

Moderate Non-exhaustive libraries/ list

The libraries list uses … (ellipsis) but doesn't call this out explicitly. An agent may treat the list as complete and fail to discover other libraries (e.g. permissions, analytics). Suggest adding: "(partial — see lib/libraries/ for the full list)".

Minor Missing UI-layer error surfacing guidance

The Either section covers cross-layer usage but stops at the domain boundary. Agents have no guidance on mapping Failure to user-facing messages in the presentation layer. A single line like "presentation maps Failure to localized UI messages; see ErrorMessageMapper or the l10n ARB files" would close the loop.

Minor No PowerSync / offline-first note

The intro mentions PowerSync but the body doesn't. Agents could write data sources that assume live network availability and inadvertently break offline behaviour. Even one line — "PowerSync handles offline sync; data sources must not assume live network" — would prevent a class of bugs.

Minor Stale CI path risk

The hardcoded reference .github/workflows/run_c_check.yml will silently go stale if the workflow file is renamed. Low risk, but worth noting for future maintainers.

Minor No AppBootstrap extension guidance

main.dart is described as building AppBootstrap but agents have no guidance on how to extend it when adding a new environment-level config. A pointer to the bootstrap class or a one-line pattern would help.

Security Considerations

None — documentation-only change. No executable code, no secrets, no dependency changes.

Test Coverage

The PR includes a manual test plan: verify commands are accurate, confirm the enforced vs. disabled lint split matches analysis_options.yaml, and confirm the feature list is complete. No automated test is applicable or needed for a documentation file.
