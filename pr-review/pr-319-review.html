Overview

This PR centralises the error-to-failure mapping that was previously handled inline by a private _handleError method inside ProjectSettingRepositoryImpl. The 59-line method is replaced by calls to ProjectErrorMapper.toFailure(error), which already covers the full range of exception types (timeout, socket, network, not-found, server, Postgres codes, permission denied). The repository's catch blocks now conditionally call _logger.error() only for unexpectedError cases, aligning with Sentry quota guidance. Tests in both project_error_mapper_test.dart and project_setting_repository_impl_test.dart are updated accordingly.

The refactor goal is sound and the mapper itself is well-implemented. Two issues need addressing before merge.

Issues Found

HIGH   #1 — Stream onError path: all logging silently dropped

In the realtime subscription .listen(onError:) block, the PR removes the existing _logger.warning(...) call without adding any replacement — neither conditional logging nor a ProjectErrorMapper call. Unlike the three CRUD catch-blocks (which preserved logging for unexpected errors), this path is now completely silent. Realtime subscription failures will be swallowed server-side with no Sentry event and no log entry.

File: lib/libraries/project/data/repositories/project_setting_repository_impl.dart, stream onError callback (~line 131).

Fix: Apply the same pattern used in the CRUD catch-blocks:

onError: (Object error, StackTrace stackTrace) {
  final failure = ProjectErrorMapper.toFailure(error);
  if (failure.errorType == ProjectErrorType.unexpectedError) {
    _logger.error(
      'Unexpected error watching project setting changes for projectId: $projectId',
      error,
      stackTrace,
    );
  }
  final controller = _settingControllers[projectId];
  if (controller?.isClosed == false) {
    controller?.addError(error, stackTrace);
  }
}
MEDIUM   #2 — Expected errors (notFound, timeout, connection, parsing) are now completely silent

The old _handleError emitted a _logger.warning() for every handled exception type. The new code only logs when errorType == unexpectedError. All "expected" failures — not-found, timeout, connection, parsing, database — leave no server-side trace. Investigating spikes of these errors in production will be difficult without log data.

Fix: Add a warning-level log for non-unexpected failures to preserve operational visibility:

final failure = ProjectErrorMapper.toFailure(error);
if (failure.errorType == ProjectErrorType.unexpectedError) {
  _logger.error('Unexpected error while getting project setting for projectId: $projectId', error, stackTrace);
} else {
  _logger.warning('Error (${failure.errorType.name}) getting project setting for projectId: $projectId', error, stackTrace);
}
return Left(failure);

Suggestions (non-blocking)

SUGGESTION   #3 — Pass StackTrace into ProjectErrorMapper

toFailure(Object error) currently discards the stack trace entirely. All catch-blocks have catch (error, stackTrace) but pass only error. Consider accepting an optional StackTrace? parameter to allow the mapper to attach it to a Failure object or log it internally in the future without requiring call-site changes. 

static ProjectFailure toFailure(Object error, [StackTrace? stackTrace]) { ... }
SUGGESTION   #4 — Add stream error path test coverage

The repository test suite covers fetchProjectSetting, updateProject, and deleteProject error paths but has no test for the realtime watchProjectSetting stream emitting an error. A test verifying that the stream controller receives the error (and logging behaves correctly) would close this gap.
